# 需求.md
# iOS/iPadOS 局域网群组发现与实时通讯 App（go-libp2p + gomobile）需求与开发计划（V1/V2）
go-libp2p: https://github.com/libp2p/go-libp2p

## 1. 项目概述
在 iOS / iPadOS 设备上实现一个实时通讯 App，使安装 App 的设备能够相互连接并进行群组内实时通信。底层网络使用 go-libp2p，采用 gomobile bind 封装为 iOS 可用的 xcframework，由 Swift 调用。


### 1.1 核心能力
- 创建群组、发现群组、加入/离开群组
- 群组内实时发送/接收数据（文本/二进制）
- 显示群组成员（在线成员列表）
- V1 仅支持局域网；V2 增加跨网发现与通信

---

## 2. 版本规划与范围

### 2.1 V1（MVP）：仅局域网（同一 Wi‑Fi/同网段）
**目标：** 在局域网内稳定可用，完成端到端群组通讯闭环。

#### V1 In Scope
1. 节点启动/停止（PeerID 持久化）
2. mDNS（Bonjour）发现局域网内 peers
3. 群组创建
4. 群组发现（基于“向已发现 peer 拉取群组列表”）
5. 加入/离开群组
6. 群组消息（PubSub topic）
7. 成员在线列表（presence 心跳 + TTL）
8. 基础安全：传输加密（Noise/TLS），消息大小/频率限制
9. iOS 本地网络权限提示与错误处理
10. 基础诊断信息（PeerID、连接数、订阅 topics、消息计数）

#### V1 Out of Scope
- 跨网发现、跨网通信（蜂窝/异地）
- 离线消息、历史消息
- 文件传输（可后续）
- 完整的端到端群组加密（可后续）

---

### 2.2 V2：跨网发现与跨网通信（增强版）
**目标：** 不同网络间可发现群组并稳定通信，NAT/蜂窝环境有兜底方案。

#### V2 计划能力
1. 引入公网 Relay（Circuit Relay v2）用于 NAT 兜底
2. 引入 Bootstrap 节点用于跨网 peer 发现/连接引导
3. 跨网发现方案（二选一或组合）：
   - 方案 A：libp2p DHT（更去中心化，需调参控制资源）
   - 方案 B：弱中心化目录服务（HTTPS，仅存群组元数据/连接信息，不存消息）
4. 网络切换恢复（Wi‑Fi ↔ 蜂窝）稳定性优化
5. 安全增强：更严格的入群鉴权、滥用控制、可选群组密钥体系

## 3. 总体架构

### 3.1 模块划分
1. **App 层（Swift）**
   - UI：群组列表、发现页、聊天页、成员页、设置页
   - 状态管理：节点状态、群组状态、消息流、成员列表
   - 调用 P2PBridge（Go Framework）API

2. **P2PBridge（Go：go-libp2p）**
   - Host 创建与网络配置
   - 发现：mDNS（V1）
   - 群组：PubSub（GossipSub）消息与 presence
   - Stream 协议：群组列表查询等
   - 事件回调：Go -> Swift

3. **（V2）公网基础设施**
   - Relay 节点、Bootstrap 节点
   - （可选）目录服务

## 4. 协议与命名规范

### 4.1 命名约定
- `appNS`: `myapp`（可替换为产品名）
- `env`: `prod | staging | dev`
- PubSub Topic 前缀：`/<appNS>/<env>/...`
- Stream Protocol 前缀：`/<appNS>/<env>/.../1.0.0`

### 4.2 PubSub Topics
1. 群组消息：
- Topic：`/<appNS>/<env>/group/<groupID>/msg`

2. 群组 presence：
- Topic：`/<appNS>/<env>/group/<groupID>/presence`
- 心跳间隔：5s（可配置）
- 离线 TTL：15s（可配置）

### 4.3 Stream Protocols（V1 必需）
1. 群组列表查询（发现页核心）：
- Protocol：`/<appNS>/<env>/groups/1.0.0`
- 请求：空（或带 since）
- 响应：该 peer 公开群组列表（groupID/name/是否口令/活跃时间/成员数估计）

（可选）群组元数据：
- Protocol：`/<appNS>/<env>/groupmeta/1.0.0`

## 6. 功能需求（V1）

### 6.1 节点生命周期
- 启动/停止节点
- PeerID/私钥持久化（Keychain/文件）
- 节点状态事件回调（连接数、监听地址、错误）

### 6.2 群组管理
- 创建群组（name + 可选 passcode + public/private）
- 发现群组（局域网：mDNS -> /groups 拉取）
- 加入/离开群组（订阅/退订 topics）

### 6.3 群组消息
- 发送文本/二进制（限制最大 payload，如 64KB~256KB）
- 接收消息实时展示
- msgID 去重、防刷限流

### 6.4 成员在线列表
- presence 心跳 + TTL 维护在线成员
- 成员列表展示（昵称、PeerID、最后心跳）

### 6.5 iOS 权限与提示
- 本地网络权限（Local Network Usage Description）
- 未授权时给出引导提示
- 明确提示：V1 仅支持局域网

## 8. Go Bridge（gomobile）接口需求

### 8.1 导出对象与方法（建议）
- `Node = NewNode(listener)`
- `Start(configJSON) -> nodeInfoJSON`
- `Stop()`
- `SetDisplayName(name)`
- `CreateGroup(name, passcode, isPublic) -> groupJSON`
- `StartDiscovery() / StopDiscovery()`
- `GetDiscoveredGroups() -> groupsJSON`（或仅事件推送）
- `JoinGroup(groupID, passcode)`
- `LeaveGroup(groupID)`
- `SendGroupMessage(groupID, data, mime) -> msgID`

### 8.2 事件回调（listener）
- `OnNodeStatus(json)`
- `OnPeerConnected(peerID) / OnPeerDisconnected(peerID)`
- `OnGroupDiscovered(groupJSON)`
- `OnGroupJoined(groupID) / OnGroupLeft(groupID)`
- `OnGroupMessage(groupID, metaJSON, data)`
- `OnPresenceUpdate(groupID, membersJSON)`
- `OnError(where, message)`

## 9. 开发计划（最终版）

> 说明：时间为建议区间；可按团队规模调整。建议以“可运行闭环”为第一优先级。

### 9.1 Milestone 0：工程与构建打通（3~5 天）
**目标**：iOS 工程能集成 P2PBridge.xcframework 并运行。

**任务**：
- 建立 Go module：`p2pbridge`（空壳 API + listener）
- gomobile init + bind 产出 framework
- 打包 xcframework（真机 arm64 + 模拟器 arm64）
- Xcode 集成与最小调用（Start/Stop）
- CI（可选）：固定 Go/Xcode 版本，自动构建产物

**交付物**：
- `P2PBridge.xcframework`
- iOS Demo：显示 PeerID、启动/停止成功

---

### 9.2 Milestone 1：局域网发现与连接（1~2 周）
**目标**：局域网内能自动发现 peer 并建立连接。

**任务（Go）**：
- libp2p Host：TCP + 安全通道 + muxer
- mDNS discovery：发现 peers 并触发连接尝试
- 连接管理：上限、退避重试、连接事件回调
- 诊断信息：已连接 peers 数、监听地址等

**任务（iOS）**：
- 本地网络权限处理与提示
- 诊断页（PeerID、连接数）

**验收**：
- 两台真机同 Wi‑Fi，能互相发现并显示“已连接/可连接”

---

### 9.3 Milestone 2：群组发现/创建/加入（1~2 周）
**目标**：“群组”闭环跑通（发现 → 加入 → 离开）。

**任务（Go）**：
- 群组数据结构与本地管理（公开群组列表）
- Stream 协议：`/groups/1.0.0` handler + client
- 发现页数据聚合与去重：从多个 peer 拉取 groups
- Join/Leave：订阅/退订 group topics

**任务（iOS）**：
- 群组列表页（已加入）
- 发现页（显示可加入群组）
- 创建群组 UI
- 加入口令 UI（先占位；V1 可不强制）

**验收**：
- A 创建群组，B 在发现页 10 秒内看到并加入；离开后不再接收消息

---

### 9.4 Milestone 3：群聊消息与成员在线列表（1~2 周）
**目标**：群组内实时消息 + 成员列表可用。

**任务（Go）**：
- PubSub（GossipSub）：
  - msg topic publish/subscribe
  - presence topic publish/subscribe
- 消息 envelope + msgID 去重
- presence 心跳与 TTL 成员表维护
- 限流与最大消息大小限制

**任务（iOS）**：
- 聊天页（消息列表 + 输入发送）
- 成员页（在线成员展示）
- UI 状态与回调线程安全（主线程更新）

**验收**：
- 两台设备加入同群，互发消息；成员列表在 10 秒内稳定一致

---

### 9.5 Milestone 4：稳定性、体验与发布准备（1~2 周）
**目标**：提升稳定性与可维护性，具备 TestFlight 发布质量。

**任务**：
- 网络异常处理（Wi‑Fi 断开/切换）下的提示与恢复策略（尽力）
- 日志系统与导出（便于定位问题）
- 参数可配置（presence interval/TTL/连接上限）

- 基础自动化测试（可选）：协议序列化、去重、TTL 逻辑

**验收**：
- 连续运行 30 分钟局域网群聊稳定；异常场景有明确提示不崩溃

## 10. V2 开发计划（跨网，计划项）

### 10.1 Milestone V2-1：Relay + Bootstrap（2~4 周）
**目标**：实现 NAT 穿透与公网中继，支撑跨局域网通信。

**任务（服务端）**：
- 部署公网 **Relay 节点**（≥2 个，不同地域/可用区），启用 libp2p Circuit Relay v2
- 部署公网 **Bootstrap 节点**（≥1 个），提供稳定的节点发现入口
- 配置节点监控与告警（CPU/内存/带宽/连接数）

**任务（客户端）**：
- 集成并启用 libp2p Relay Client
- 启动时优先连接 Bootstrap 节点
- 实现自动回退机制：直连 → NAT 穿透 → Relay 中继
- 在 NodeInfo 中暴露当前连接类型（direct / relayed）

**验收标准**：
- 两台设备分别位于不同 Wi‑Fi 网络（如 4G/家庭宽带）
- 双方可通过 Relay 加入同一群组并正常收发消息
- 消息传输稳定，无持续丢包或连接中断

---

### 10.2 Milestone V2-2：跨网发现（2~4 周）
**目标**：解决跨局域网环境下的群组与 Peer 发现问题。

**方案 A：DHT（去中心化）**
- 集成 libp2p Kademlia DHT
- 实现 Group 记录的 DHT 存储与查询（基于 GroupID）
- 实现 Peer 路由发现
- 严格控制 DHT 参数（桶大小、并发查询数、刷新周期）以降低资源消耗

**方案 B：目录服务（中心化元数据，推荐初期采用）**
- 设计 RESTful HTTPS 目录服务 API：
  - `POST /api/v1/groups/announce`（群组注册/心跳）
  - `GET /api/v1/groups/discover`（拉取群组列表）
- 客户端定期向目录服务上报群组信息（GroupID、名称、公钥指纹、TTL）
- 客户端通过目录服务获取远端群组列表，再通过 P2P 建立连接
- P2P 消息传输不经过目录服务

**决策建议**：
- V2 早期：优先实现 **目录服务**（开发快、可控性强、易排查问题）
- 后续迭代：视规模与去中心化需求再评估引入 DHT

**验收标准**：
- 跨网环境下，客户端可在 15 秒内发现远端网络发布的群组
- 发现结果包含必要的鉴权信息（如是否需要口令）

---

### 10.3 Milestone V2-3：安全增强与优化（2~4 周）
**目标**：提升跨网通信的安全性与抗滥用能力。

**任务（安全）**：
- 实现入群鉴权：**Challenge/Response**
  - 群组创建者生成随机 Challenge
  - 申请入群者使用群组密钥（或密码派生密钥）签名响应
  - 验证通过后方可订阅群组 Topic
- 实现群组级对称加密（可选，但强烈建议）：
  - 基于 GroupID + Passcode 派生 Group Key
  - 所有 PubSub 消息使用 AES-GCM 或 ChaCha20-Poly1305 加密
  - 确保即使 Relay 节点也无法解密消息内容

**任务（反滥用）**：
- 实现连接级与消息级限流（Rate Limiting）
- 实现 Peer 黑名单/禁言机制
- 实现基础反垃圾策略（重复消息、高频消息检测）

**验收标准**：
- 无法提供正确凭证的节点无法加入群组或窃听消息
- 中继节点仅转发密文，无法获取明文内容
- 高频刷屏行为被有效抑制

## 11. V1 验收标准（最终）

**环境**：两台 iOS 真机，连接至同一 Wi‑Fi 局域网。

| 编号 | 验收项 | 预期结果 |
| :--- | :--- | :--- |
| 1 | 节点发现 | 通过 mDNS 互相发现对方 PeerID |
| 2 | 群组创建与发现 | A 创建群组，B 在发现页 10 秒内可见 |
| 3 | 群组加入 | B 成功加入群组，双方均收到加入通知 |
| 4 | 消息延迟 | 群组内互发文本消息，端到端延迟典型值 < 300ms |
| 5 | 成员列表一致性 | 成员在线状态在 10 秒内收敛并保持一致 |
| 6 | 退群隔离 | 离开群组后，不再接收到该群的任何消息 |
| 7 | 身份持久化 | App 完全重启（杀进程/重启手机）后，PeerID 保持不变 |
| 8 | 权限处理 | 未授权“本地网络”权限时，有明确的弹窗提示与系统设置引导 |

## 12. 风险与对策（V1）

| 风险点 | 影响 | 对策 |
| :--- | :--- | :--- |
| **iOS 本地网络权限 / Bonjour 限制** | mDNS 发现失败，导致无法组网 | - 首次进入发现页时主动触发权限弹窗<br>- 若授权被拒，展示专用引导页，说明为何需要该权限及如何前往系统设置开启 |
| **mDNS 在复杂网络下不稳定** | 部分路由器/企业网络下无法发现 Peer | - V1 产品层面明确标注“仅支持局域网”<br>- 在诊断页提供手动添加 Peer（IP/端口）的应急入口<br>- V2 通过 Relay/Bootstrap/目录服务解决 |
| **PubSub 资源消耗过高** | 低端设备发热、耗电、卡顿 | - 硬性限制：最大连接数（40）、Topic 数（每个群 2 个）、单条消息大小（64–256KB）<br>- 消息频率限流（如 10 条/秒/群）<br>- 后台时暂停心跳与消息广播 |
| **iOS 后台模式限制** | App 切后台后连接中断 | - V1 明确不支持后台常驻<br>- 切后台后主动关闭 libp2p Host，前台恢复后重连 |
| **PeerID 丢失/重置** | 用户身份变化，历史消息关联失效 | - 使用 iOS Keychain 存储私钥，开启 `kSecAttrAccessibleAfterFirstUnlock`<br>- 提供“重置身份”的显式入口，普通退出不清除 Keychain |
